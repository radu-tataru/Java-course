1. What are Packages?

Packages are namespaces that organize related classes and interfaces into groups. Think of them as folders for your Java code.

Why Use Packages?

  • Organization: Group related classes together (e.g., all database classes in com.example.database)
  • Namespace Management: Avoid naming conflicts (two classes named User can exist in different packages)
  • Access Control: Use package-private access to restrict visibility
  • Modularity: Create reusable components that can be distributed as libraries

Java's Built-in Packages

Package Description Common Classes
java.lang Fundamental classes (auto-imported) String, Math, System, Integer
java.util Utility classes and collections ArrayList, HashMap, Scanner, Date
java.io Input/output operations File, FileReader, BufferedReader
java.nio New I/O (non-blocking I/O) Path, Files, Charset
java.net Networking operations URL, Socket, HttpURLConnection
java.time Date and time API (Java 8+) LocalDate, LocalTime, Duration
Note: The java.lang package is automatically imported in every Java file. You don't need to explicitly import classes like String, Math, or System.

2. Creating Packages

Package Declaration

The package statement must be the first line in your Java file (except for comments).

package com.example.models;

public class User {
    private String name;
    private String email;

    // Constructor, getters, setters...
}

Package Naming Conventions

  • All lowercase: Package names should be in lowercase to avoid conflicts with class names
  • Reverse domain name: Start with your reversed domain name (e.g., com.company.project)
  • Descriptive names: Use meaningful names that describe the content (models, services, utils)
Good Package Names Bad Package Names
com.example.app.models com.Example.App.Models (uppercase)
org.project.database myPackage (not descriptive)
io.company.utils package1 (meaningless)

Folder Structure

Packages correspond to folder structures in your file system:

src/
└── com/
    └── example/
        └── app/
            ├── models/
            │   ├── User.java          (package com.example.app.models)
            │   └── Product.java       (package com.example.app.models)
            ├── services/
            │   ├── UserService.java   (package com.example.app.services)
            │   └── OrderService.java  (package com.example.app.services)
            └── Main.java              (package com.example.app)
Warning: The folder structure must exactly match the package declaration. If your package is com.example.app, the file must be in com/example/app/ folder.

3. Importing Classes

Single Class Import

Import a specific class from a package:

import java.util.ArrayList;
import java.util.Scanner;

public class Main {
    public static void main(String[] args) {
        ArrayList list = new ArrayList<>();
        Scanner scanner = new Scanner(System.in);
    }
}

Wildcard Import

Import all classes from a package using *:

import java.util.*;

public class Main {
    public static void main(String[] args) {
        ArrayList list = new ArrayList<>();
        HashMap map = new HashMap<>();
        Scanner scanner = new Scanner(System.in);
    }
}

Fully Qualified Names (No Import)

Use the full package path without importing:

public class Main {
    public static void main(String[] args) {
        java.util.ArrayList list = new java.util.ArrayList<>();
        java.util.Scanner scanner = new java.util.Scanner(System.in);
    }
}

When to Use Each Approach

Import Type When to Use Example
Single Class When you need only a few specific classes import java.util.ArrayList;
Wildcard When using many classes from the same package import java.util.*;
Fully Qualified To resolve naming conflicts or one-time use java.sql.Date sqlDate;
Tip: Wildcard imports (*) don't impact performance—they're resolved at compile time. However, explicit imports make your dependencies clearer.

4. Handling Naming Conflicts

When two classes have the same name but are in different packages, you'll encounter a naming conflict.

Example: java.util.Date vs java.sql.Date

// ❌ This will cause a compile error
import java.util.Date;
import java.sql.Date;  // Error: Date is already defined

public class Main {
    public static void main(String[] args) {
        Date date = new Date();  // Which Date?
    }
}

Solution 1: Use Fully Qualified Names

import java.util.Date;

public class Main {
    public static void main(String[] args) {
        Date utilDate = new Date();                    // java.util.Date
        java.sql.Date sqlDate = new java.sql.Date(0);  // java.sql.Date
    }
}

Solution 2: Import One, Qualify the Other

import java.util.*;

public class Main {
    public static void main(String[] args) {
        Date utilDate = new Date();                    // java.util.Date (imported)
        java.sql.Date sqlDate = new java.sql.Date(0);  // Fully qualified
    }
}
Warning: When in doubt, use fully qualified names for the less frequently used class to make your code's intent clear.

5. Static Imports

Static imports allow you to import static members (methods and fields) of a class directly.

Without Static Import

public class Calculator {
    public static void main(String[] args) {
        double result = Math.sqrt(16);
        double pi = Math.PI;
        double power = Math.pow(2, 3);

        System.out.println("Square root: " + result);
        System.out.println("Pi: " + pi);
        System.out.println("Power: " + power);
    }
}

With Static Import

import static java.lang.Math.*;

public class Calculator {
    public static void main(String[] args) {
        double result = sqrt(16);      // No need for Math.
        double pi = PI;                // No need for Math.
        double power = pow(2, 3);      // No need for Math.

        System.out.println("Square root: " + result);
        System.out.println("Pi: " + pi);
        System.out.println("Power: " + power);
    }
}

Static Import - Specific Members

import static java.lang.Math.PI;
import static java.lang.Math.sqrt;
import static java.lang.System.out;

public class Calculator {
    public static void main(String[] args) {
        double result = sqrt(16);
        out.println("Square root: " + result);
        out.println("Pi: " + PI);
    }
}

When to Use Static Imports

  • ✅ Mathematical constants and functions: Math.PI, Math.sqrt()
  • ✅ Frequently used static methods: Test assertions (assertEquals()), utility methods
  • ❌ Avoid overuse: Can make code less readable if the origin of methods is unclear
Tip: Static imports are commonly used in testing frameworks like JUnit to make test code more readable: import static org.junit.Assert.*;

6. Package Access Modifiers

Access modifiers control the visibility of classes, methods, and fields across packages.

Access Levels

Modifier Same Class Same Package Subclass Other Packages
public ✅ ✅ ✅ ✅
protected ✅ ✅ ✅ ❌
default (no modifier) ✅ ✅ ❌ ❌
private ✅ ❌ ❌ ❌

Example: Package-Private (Default) Access

File: com/example/models/User.java

package com.example.models;

class User {  // No modifier = package-private
    String name;  // Package-private field

    void display() {  // Package-private method
        System.out.println("User: " + name);
    }
}

File: com/example/models/UserManager.java

package com.example.models;

public class UserManager {
    public void test() {
        User user = new User();  // ✅ Accessible - same package
        user.name = "Alice";     // ✅ Accessible - same package
        user.display();          // ✅ Accessible - same package
    }
}

File: com/example/app/Main.java

package com.example.app;

import com.example.models.User;  // ❌ Compile error - User is not public

public class Main {
    public static void main(String[] args) {
        User user = new User();  // ❌ Can't access - different package
    }
}
Note: Package-private (default) access is useful for internal implementation classes that shouldn't be exposed outside the package.

7. Real-World Package Structure

Typical Multi-Layer Application

src/
└── com/
    └── example/
        └── app/
            ├── models/              # Data models (entities)
            │   ├── User.java
            │   ├── Product.java
            │   └── Order.java
            │
            ├── services/            # Business logic
            │   ├── UserService.java
            │   ├── ProductService.java
            │   └── OrderService.java
            │
            ├── repositories/        # Data access layer
            │   ├── UserRepository.java
            │   └── ProductRepository.java
            │
            ├── controllers/         # Web/API controllers
            │   ├── UserController.java
            │   └── ProductController.java
            │
            ├── utils/               # Utility classes
            │   ├── DateUtils.java
            │   └── StringUtils.java
            │
            ├── exceptions/          # Custom exceptions
            │   └── UserNotFoundException.java
            │
            └── Main.java            # Application entry point

Example: Using the Structure

Model Class:

package com.example.app.models;

public class User {
    private String id;
    private String name;
    private String email;

    // Constructors, getters, setters...
}

Service Class:

package com.example.app.services;

import com.example.app.models.User;
import com.example.app.repositories.UserRepository;

public class UserService {
    private UserRepository repository = new UserRepository();

    public User getUserById(String id) {
        return repository.findById(id);
    }
}

Main Application:

package com.example.app;

import com.example.app.models.User;
import com.example.app.services.UserService;

public class Main {
    public static void main(String[] args) {
        UserService userService = new UserService();
        User user = userService.getUserById("123");
        System.out.println("User: " + user.getName());
    }
}

Key Takeaways

Organize with Packages: Use packages to group related classes and prevent naming conflicts.
Follow Naming Conventions: Use lowercase, reverse domain name format (com.company.project).
Match Folder Structure: Package declarations must match the folder structure exactly.
Choose Import Type: Use single imports for clarity, wildcards for convenience, fully qualified names for conflicts.
Static Imports Sparingly: Use static imports for frequently used static members like Math methods.
Control Access: Use package-private (default) access for internal implementation classes.
java.lang Auto-imported: Classes like String, Math, and System don't need explicit imports.
