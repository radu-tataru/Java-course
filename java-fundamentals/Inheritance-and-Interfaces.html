Inheritance: The "extends" Relationship

Inheritance allows a class to inherit properties and methods from another class, creating an "is-a" relationship.

Key Concept: Inheritance represents a "is-a" relationship. A Car "is-a" Vehicle, so Car can extend Vehicle.

Basic Inheritance Example

// Parent class (superclass)
public class Vehicle {
    protected String brand;
    protected int year;

    public Vehicle(String brand, int year) {
        this.brand = brand;
        this.year = year;
    }

    public void start() {
        System.out.println(brand + " is starting...");
    }

    public void displayInfo() {
        System.out.println(year + " " + brand);
    }
}

// Child class (subclass)
public class Car extends Vehicle {
    private int doors;

    public Car(String brand, int year, int doors) {
        super(brand, year); // Call parent constructor
        this.doors = doors;
    }

    // Override parent method
    @Override
    public void start() {
        System.out.println("Car engine starting with key...");
    }

    // Add new method specific to Car
    public void openTrunk() {
        System.out.println("Trunk opened");
    }
}

The @Override Annotation

Best Practice: Always use @Override when overriding methods. It helps catch errors and makes your intent clear.
With @Override Without @Override
✅ Compiler verifies you're actually overriding ❌ Typos create new methods instead
✅ Clear documentation of intent ❌ Unclear if method is new or override
✅ IDE support and warnings ❌ No IDE assistance

Interfaces: The Contract Pattern

An interface defines a contract that classes must follow. It specifies what methods a class must have, but not how they work.

Key Insight: Interfaces define "can-do" relationships. A Bird "can-fly", a Fish "can-swim" - these are capabilities, not inheritance.

Interface Definition and Implementation

// Interface definition
public interface Flyable {
    void takeOff();
    void fly();
    void land();

    // Default method (Java 8+)
    default void checkWeather() {
        System.out.println("Checking weather conditions...");
    }
}

// Interface implementation
public class Bird implements Flyable {
    private String species;

    public Bird(String species) {
        this.species = species;
    }

    @Override
    public void takeOff() {
        System.out.println(species + " flaps wings and takes off");
    }

    @Override
    public void fly() {
        System.out.println(species + " is flying gracefully");
    }

    @Override
    public void land() {
        System.out.println(species + " lands softly");
    }
}

// Another implementation
public class Airplane implements Flyable {
    private String model;

    public Airplane(String model) {
        this.model = model;
    }

    @Override
    public void takeOff() {
        System.out.println(model + " accelerates down runway");
    }

    @Override
    public void fly() {
        System.out.println(model + " cruising at 30,000 feet");
    }

    @Override
    public void land() {
        System.out.println(model + " approaches runway for landing");
    }
}

Polymorphism in Action

Polymorphism allows objects of different types to be treated uniformly when they share a common interface or parent class.

Interface Polymorphism Example

public class FlightControl {
    public static void manageFlying(Flyable flyer) {
        // Works with ANY class that implements Flyable
        flyer.checkWeather();  // Default method
        flyer.takeOff();
        flyer.fly();
        flyer.land();
    }

    public static void main(String[] args) {
        // Different implementations, same interface
        Flyable bird = new Bird("Eagle");
        Flyable plane = new Airplane("Boeing 747");

        // Polymorphism - same method call, different behavior
        manageFlying(bird);   // Bird's implementation
        manageFlying(plane);  // Airplane's implementation
    }
}
Powerful Pattern: The manageFlying method doesn't need to know if it's working with a Bird or Airplane. It just knows both can fly!

Abstract Classes: The Middle Ground

Abstract classes combine features of both inheritance and interfaces. They can have both implemented methods and abstract methods that must be overridden.

// Abstract class - cannot be instantiated directly
public abstract class Animal {
    protected String name;
    protected int age;

    public Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // Concrete method - all animals have this
    public void sleep() {
        System.out.println(name + " is sleeping");
    }

    // Abstract method - each animal implements differently
    public abstract void makeSound();
    public abstract void move();
}

// Concrete implementation
public class Dog extends Animal {
    public Dog(String name, int age) {
        super(name, age);
    }

    @Override
    public void makeSound() {
        System.out.println(name + " barks: Woof!");
    }

    @Override
    public void move() {
        System.out.println(name + " runs on four legs");
    }
}

When to Use What?

Use Case Choose Example Keyword
Strong "is-a" relationship Inheritance Car extends Vehicle extends
Define capabilities/contracts Interface Bird implements Flyable implements
Share code + enforce contracts Abstract Class Dog extends Animal extends
Multiple capabilities Multiple Interfaces Duck implements Flyable, Swimmable implements
Remember: A class can extend only ONE class, but can implement MULTIPLE interfaces. This is why interfaces are often preferred for defining capabilities.

Practical Example: File Readers

Here's how these concepts apply to the file reading project you'll encounter in Project Features:

// Interface defining contract for file readers
public interface LinkReader {
    List<String> readLinks(String filePath) throws IOException;
}

// Concrete implementations
public class TxtLinkReader implements LinkReader {
    @Override
    public List<String> readLinks(String filePath) throws IOException {
        List<String> links = new ArrayList<>();
        // TXT-specific reading logic
        return links;
    }
}

public class JsonLinkReader implements LinkReader {
    @Override
    public List<String> readLinks(String filePath) throws IOException {
        List<String> links = new ArrayList<>();
        // JSON-specific reading logic
        return links;
    }
}

// Polymorphic usage
public class FileProcessor {
    public void processFile(LinkReader reader, String filePath) {
        try {
            List<String> links = reader.readLinks(filePath);
            // Process links regardless of file type
        } catch (IOException e) {
            System.err.println("Error reading file: " + e.getMessage());
        }
    }
}
Project Connection: This exact pattern is used in Project Features Step 3. The interface allows treating all file readers the same way, regardless of file type!

Key Takeaways

Inheritance (extends)
Creates "is-a" relationships and shares code between related classes
Interfaces (implements)
Define contracts and "can-do" capabilities for unrelated classes
@Override annotation
Always use it for safety and clarity when overriding methods
Polymorphism
Treat different objects uniformly through shared interfaces or parent classes
