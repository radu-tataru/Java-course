Architecture Requirements

1. BasePage Structure

Your BasePage should include:

public abstract class BasePage {
    protected WebDriver driver;
    protected WebDriverWait wait;

    public BasePage(WebDriver driver) {
        this.driver = driver;
        this.wait = new WebDriverWait(driver, Duration.ofSeconds(10));
    }

    // Common methods all pages need:
    protected void click(By locator) { /* ... */ }
    protected void type(By locator, String text) { /* ... */ }
    protected String getText(By locator) { /* ... */ }
    protected boolean isElementVisible(By locator) { /* ... */ }
    protected void waitForElement(By locator) { /* ... */ }
    protected void takeScreenshot(String filename) { /* ... */ }

    public abstract String getPageTitle();
    public abstract boolean isPageLoaded();
}

2. Page Object Structure

Each page class should follow this pattern:

public class WikipediaArticlePage extends BasePage {

    // Locators (private and organized by section)
    private By searchInput = By.id("searchInput");
    private By searchButton = By.cssSelector("button[type='submit']");
    private By articleTitle = By.id("firstHeading");
    private By randomArticleLink = By.id("n-randompage");

    // Constructor
    public WikipediaArticlePage(WebDriver driver) {
        super(driver);
    }

    // Page-specific actions (public methods)
    public void searchForArticle(String searchTerm) {
        type(searchInput, searchTerm);
        click(searchButton);
    }

    public String getArticleTitle() {
        return getText(articleTitle);
    }

    public void navigateToRandomArticle() {
        click(randomArticleLink);
    }

    // Verification methods
    @Override
    public String getPageTitle() {
        return driver.getTitle();
    }

    @Override
    public boolean isPageLoaded() {
        return isElementVisible(articleTitle);
    }
}

3. Test Structure

Your test class should look like:

public class WikipediaTest {
    private WebDriver driver;
    private WikipediaArticlePage articlePage;

    @BeforeEach
    public void setUp() {
        WebDriverManager.chromedriver().setup();
        driver = new ChromeDriver();
        driver.manage().window().maximize();
        driver.get("https://www.wikipedia.org");

        articlePage = new WikipediaArticlePage(driver);
    }

    @Test
    @DisplayName("Search for an article and verify title")
    public void testArticleSearch() {
        articlePage.searchForArticle("Selenium (software)");

        assertTrue(articlePage.isPageLoaded(),
            "Article page should be loaded");
        assertEquals("Selenium (software)", articlePage.getArticleTitle(),
            "Article title should match search term");
    }

    @AfterEach
    public void tearDown() {
        if (driver != null) {
            driver.quit();
        }
    }
}

Your solution must include:

  2. At least 2 page objects (3+ for advanced challenge)
  3. Proper encapsulation - locators private, actions public
  4. Meaningful method names that describe actions from user perspective
  5. At least 5 test scenarios (8+ for advanced)
  6. All tests passing with proper assertions
  7. Screenshot capability for failed tests
  8. README.md explaining your design decisions

Code Quality Checklist

  • ✅ DRY Principle: No duplicated code across page objects
  • ✅ Single Responsibility: Each page class represents one page/component
  • ✅ Explicit Waits: Proper WebDriverWait usage, no Thread.sleep()
  • ✅ Clean Locators: Well-organized, descriptive locator variables
  • ✅ Error Handling: Graceful handling of element not found scenarios
  • ✅ Test Independence: Each test can run independently
  • ✅ Readable Tests: Tests read like user stories

Before & After: The Power of Page Objects

❌ Before: Without Page Objects (Messy)

@Test
public void testWikipediaSearch() {
    driver.get("https://www.wikipedia.org");

    WebElement searchBox = driver.findElement(By.id("searchInput"));
    searchBox.sendKeys("Selenium");

    WebElement searchBtn = driver.findElement(By.cssSelector("button[type='submit']"));
    searchBtn.click();

    WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
    wait.until(ExpectedConditions.presenceOfElementLocated(By.id("firstHeading")));

    String title = driver.findElement(By.id("firstHeading")).getText();
    assertEquals("Selenium", title);
}

@Test
public void testWikipediaRandomArticle() {
    driver.get("https://www.wikipedia.org");

    // Same locators repeated!
    WebElement randomLink = driver.findElement(By.id("n-randompage"));
    randomLink.click();

    WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
    wait.until(ExpectedConditions.presenceOfElementLocated(By.id("firstHeading")));

    String title = driver.findElement(By.id("firstHeading")).getText();
    assertNotNull(title);
}

✅ After: With Page Objects (Clean & Maintainable)

@Test
@DisplayName("Search for article and verify title")
public void testWikipediaSearch() {
    articlePage.searchForArticle("Selenium");

    assertTrue(articlePage.isPageLoaded());
    assertEquals("Selenium", articlePage.getArticleTitle());
}

@Test
@DisplayName("Navigate to random article")
public void testWikipediaRandomArticle() {
    articlePage.navigateToRandomArticle();

    assertTrue(articlePage.isPageLoaded());
    assertNotNull(articlePage.getArticleTitle());
}
Notice the difference:
  • Tests are dramatically shorter and easier to read
  • Locators are defined once in the page object
  • If Wikipedia changes their UI, you update one place
  • Tests read like user actions, not technical code
  • New team members can understand tests immediately

Tips & Best Practices

🎯 Page Object Design Tips

  • One page class per page: Don't mix multiple pages in one class
  • Actions return page objects: Enable method chaining (e.g., loginPage.login().verifyDashboard())
  • Use descriptive method names: clickLoginButton() not click()
  • Keep locators private: Tests should never access locators directly
  • Group related elements: Use comments to organize locators by section

⏰ Waiting Strategies

// ✅ GOOD: Explicit wait in BasePage
protected void waitForElement(By locator) {
    wait.until(ExpectedConditions.visibilityOfElementLocated(locator));
}

// ❌ BAD: Hard-coded waits
Thread.sleep(5000); // Don't do this!

🧪 Test Organization

  • Use @BeforeEach: Initialize WebDriver and page objects
  • Use @AfterEach: Always quit WebDriver to free resources
  • Use @DisplayName: Make test purposes crystal clear
  • One assertion per test: Or group related assertions

🐛 Common Pitfalls to Avoid

  • ❌ Using Thread.sleep(): Use explicit waits instead
  • ❌ Tests depending on each other: Each test should be independent
  • ❌ Hard-coded test data: Use variables or data providers
  • ❌ Mixing assertions and actions: Keep page objects focused on actions
  • ❌ Exposing WebDriver: Keep driver private in BasePage
