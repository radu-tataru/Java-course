Understanding Log4j 2 Architecture

Log4j 2 has three core components that work together:

📝 1. Loggers - What You Write Code Against

Loggers are the entry point for your code. You get a logger instance and call methods like logger.info(), logger.error(), etc.

import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;

public class MyTest {
    private static final Logger logger = LogManager.getLogger(MyTest.class);

    public void runTest() {
        logger.info("Test started");
        logger.debug("Debug details here");
        logger.error("Something failed!");
    }
}

📊 2. Log Levels - Controlling Verbosity

Log4j 2 has six standard log levels (from least to most severe):

Level Purpose Example Use Case
TRACE Very detailed diagnostic Method entry/exit, variable values at each step
DEBUG Detailed debugging info "Opening URL: https://...", "Element found: username_field"
INFO Important business events "Test scenario started", "Login successful", "Report generated"
WARN Potentially harmful situations "Element took 15s to load (slow)", "Deprecated API used"
ERROR Error events "Test failed: element not found", "Screenshot capture failed"
FATAL Critical failures "WebDriver crashed", "Database connection lost"
Pro Tip: Set level to INFO in production, DEBUG during test development, and TRACE only when investigating specific issues. This controls log file size and readability.

📤 3. Appenders - Where Logs Go

Appenders define output destinations. Common types:

  • ConsoleAppender: Writes to terminal (stdout/stderr)
  • FileAppender: Writes to a single file
  • RollingFileAppender: Creates new files when size/time limits hit (prevents huge files)
  • AsyncAppender: Non-blocking logging for high-performance apps

You can have multiple appenders! Common pattern: Console for immediate feedback + RollingFile for persistent history.

Adding Log4j 2 to Your Project

Step 1: Update Maven Dependencies

Add Log4j 2 dependencies to your pom.xml:

<dependencies>
    <!-- Log4j 2 Core -->
    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-core</artifactId>
        <version>2.20.0</version>
    </dependency>

    <!-- Log4j 2 API -->
    <dependency>
        <groupId>org.apache.logging.log4j</groupId>
        <artifactId>log4j-api</artifactId>
        <version>2.20.0</version>
    </dependency>

    <!-- Existing dependencies (Selenium, Cucumber, ExtentReports) remain -->
</dependencies>
Version Note: Log4j 2.20.0 (2023) is stable and production-ready. Always use version 2.17.0+ to avoid security vulnerabilities found in older versions.

Creating the Log4j Configuration File

Step 2: Create log4j2.xml

Create this file in src/main/resources/log4j2.xml:

<?xml version="1.0" encoding="UTF-8"?>
<Configuration status="WARN">
    <Properties>
        <!-- Define reusable properties -->
        <Property name="log-path">logs</Property>
        <Property name="log-pattern">%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger{36} - %msg%n</Property>
    </Properties>

    <Appenders>
        <!-- Console Appender - for immediate feedback -->
        <Console name="Console" target="SYSTEM_OUT">
            <PatternLayout pattern="${log-pattern}"/>
        </Console>

        <!-- File Appender - all logs in one file -->
        <File name="FileLogger" fileName="${log-path}/app.log" append="true">
            <PatternLayout pattern="${log-pattern}"/>
        </File>

        <!-- Rolling File Appender - creates new file daily and when size exceeds 10MB -->
        <RollingFile name="RollingFileLogger"
                     fileName="${log-path}/test-automation.log"
                     filePattern="${log-path}/test-automation-%d{yyyy-MM-dd}-%i.log">
            <PatternLayout pattern="${log-pattern}"/>
            <Policies>
                <TimeBasedTriggeringPolicy interval="1" modulate="true"/>
                <SizeBasedTriggeringPolicy size="10MB"/>
            </Policies>
            <DefaultRolloverStrategy max="30"/>
        </RollingFile>
    </Appenders>

    <Loggers>
        <!-- Root logger - applies to all classes -->
        <Root level="info">
            <AppenderRef ref="Console"/>
            <AppenderRef ref="RollingFileLogger"/>
        </Root>

        <!-- Package-specific loggers - more control -->
        <Logger name="com.example.app" level="debug" additivity="false">
            <AppenderRef ref="Console"/>
            <AppenderRef ref="RollingFileLogger"/>
        </Logger>

        <!-- Selenium WebDriver - reduce noise -->
        <Logger name="org.openqa.selenium" level="warn" additivity="false">
            <AppenderRef ref="Console"/>
        </Logger>
    </Loggers>
</Configuration>

Understanding the Configuration

Properties Section
Defines reusable variables like log directory and output format pattern
Appenders Section
Configures output destinations: Console + RollingFile for organized logging
Loggers Section
Controls verbosity per package: DEBUG for your code, WARN for noisy libraries

After: Log4j 2 Logger

package com.example.app;

import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;

public class Main {
    // Get logger for this class
    private static final Logger logger = LogManager.getLogger(Main.class);

    public static void main(String[] args) {
        logger.info("Application started");

        // Different log levels
        logger.info("Opening link: https://www.google.com");
        logger.debug("Browser instance created successfully");
        logger.error("Failed to open link: https://invalid.url");

        logger.info("Application finished");
    }
}
Much Simpler! No more custom Logger class, no getInstance() calls, no manual file handling. Log4j manages everything through configuration.

Integrating Log4j with Your Test Framework

Page Object Model with Log4j (Step 7 Enhanced)

package com.example.app.pages;

import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.WebElement;
import org.openqa.selenium.support.FindBy;
import org.openqa.selenium.support.PageFactory;

public class GoogleHomePage {
    private static final Logger logger = LogManager.getLogger(GoogleHomePage.class);
    private WebDriver driver;

    @FindBy(name = "q")
    private WebElement searchBox;

    @FindBy(name = "btnK")
    private WebElement searchButton;

    public GoogleHomePage(WebDriver driver) {
        this.driver = driver;
        PageFactory.initElements(driver, this);
        logger.info("GoogleHomePage initialized");
    }

    public void searchFor(String query) {
        logger.debug("Searching for: {}", query);
        try {
            searchBox.sendKeys(query);
            searchButton.click();
            logger.info("Search completed successfully for: {}", query);
        } catch (Exception e) {
            logger.error("Search failed for: {}", query, e);
            throw e;
        }
    }

    public String getTitle() {
        String title = driver.getTitle();
        logger.debug("Page title: {}", title);
        return title;
    }
}

Test Class with Log4j

package com.example.app.tests;

import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;
import org.junit.jupiter.api.*;
import org.openqa.selenium.WebDriver;
import org.openqa.selenium.chrome.ChromeDriver;
import com.example.app.pages.GoogleHomePage;

public class SearchTest {
    private static final Logger logger = LogManager.getLogger(SearchTest.class);
    private WebDriver driver;
    private GoogleHomePage homePage;

    @BeforeEach
    public void setup() {
        logger.info("=== Test Setup Started ===");
        driver = new ChromeDriver();
        driver.manage().window().maximize();
        logger.debug("WebDriver initialized: Chrome");

        driver.get("https://www.google.com");
        homePage = new GoogleHomePage(driver);
        logger.info("=== Test Setup Complete ===");
    }

    @Test
    @DisplayName("Verify Google Search Functionality")
    public void testGoogleSearch() {
        logger.info("TEST STARTED: testGoogleSearch");

        homePage.searchFor("Selenium WebDriver");

        String title = homePage.getTitle();
        logger.info("Verification: Page title = {}", title);

        Assertions.assertTrue(title.contains("Selenium"),
            "Title should contain 'Selenium'");

        logger.info("TEST PASSED: testGoogleSearch");
    }

    @AfterEach
    public void teardown() {
        logger.info("=== Test Teardown Started ===");
        if (driver != null) {
            driver.quit();
            logger.debug("WebDriver closed successfully");
        }
        logger.info("=== Test Teardown Complete ===");
    }
}

Log4j Best Practices for QA Automation

✅ DO - Professional Logging Practices

Use Appropriate Log Levels
INFO for test milestones, DEBUG for detailed steps, ERROR for failures
Log Method Entry/Exit
Helps trace test execution flow, especially in complex scenarios
Include Context in Messages
"Login failed for user: john@example.com" better than "Login failed"
Log Exceptions with Stack Traces
logger.error("Test failed", exception) captures full error details
Use Parameterized Logging
logger.info("User {} logged in", username) is more efficient than string concatenation

❌ DON'T - Common Logging Mistakes

Don't Log Sensitive Data
Never log passwords, API keys, credit cards, or PII
Don't Over-Log
Logging every variable slows tests and creates noise
Don't Use System.out.println
Always use logger - it's configurable and can be turned off
Don't Ignore Log Levels
Using only INFO defeats the purpose of having multiple levels
Security Note: Be especially careful in production environments. Use different configurations for dev/staging/prod to control what gets logged.
