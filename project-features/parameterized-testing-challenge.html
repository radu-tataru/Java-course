Why Parameterized Testing Matters

Real-World Impact: In professional QA automation, you'll often need to test the same functionality with different inputs: valid/invalid data, boundary values, multiple user roles, different browsers, etc. Parameterized tests make this elegant and maintainable.

From Repetitive to Elegant

‚ùå Without Parameterized Tests (Repetitive):

@Test
void testValidAdminLogin() {
    boolean result = login("admin", "password123");
    assertTrue(result, "Admin should login successfully");
}

@Test
void testValidUserLogin() {
    boolean result = login("user", "userpass");
    assertTrue(result, "User should login successfully");
}

@Test
void testInvalidPassword() {
    boolean result = login("admin", "wrongpass");
    assertFalse(result, "Invalid password should fail");
}

@Test
void testEmptyUsername() {
    boolean result = login("", "password");
    assertFalse(result, "Empty username should fail");
}

// ... potentially 20+ more similar tests!

‚úÖ With Parameterized Tests (Elegant):

@ParameterizedTest
@CsvSource({
    "admin, password123, true, Admin login should succeed",
    "user, userpass, true, User login should succeed",
    "admin, wrongpass, false, Invalid password should fail",
    "'', password, false, Empty username should fail",
    "admin, '', false, Empty password should fail",
    "guest, guest123, true, Guest login should succeed"
})
void testLogin(String username, String password, boolean expected, String scenario) {
    boolean result = login(username, password);
    assertEquals(expected, result, scenario);
}

// One method, 6+ test cases automatically executed!

JUnit 5 Parameterized Testing Quick Reference

Common Annotations

@CsvSource - Inline Data

@ParameterizedTest
@CsvSource({
    "apple, 1",
    "banana, 2",
    "'quoted string', 3"
})
void test(String fruit, int count) {
    // Test logic
}

@ValueSource - Single Values

@ParameterizedTest
@ValueSource(strings = {"apple", "banana", "orange"})
void testFruits(String fruit) {
    assertNotNull(fruit);
}

@CsvFileSource - External File

@ParameterizedTest
@CsvFileSource(
    resources = "/test-data.csv",
    numLinesToSkip = 1
)
void test(String input, String expected) {
    // Test logic
}

@MethodSource - Complex Objects

@ParameterizedTest
@MethodSource("provideTestData")
void test(User user, boolean valid) {
    // Test logic
}

static Stream<Arguments> provideTestData() {
    return Stream.of(
        Arguments.of(new User("john"), true),
        Arguments.of(new User(""), false)
    );
}

Tips & Best Practices

‚úÖ DO - Professional Practices

Use Descriptive Parameter Names
void test(String username, String password, boolean expected) is better than void test(String a, String b, boolean c)
Add Display Names
Use @DisplayName("Login Validation Tests") for readable output
Include Scenario Description
Add a parameter for test description: "admin, pass123, true, Valid admin login"
Group Related Tests
Use @Nested classes to organize parameterized tests by feature

‚ùå DON'T - Common Mistakes

Don't Mix Too Many Scenarios
Keep related scenarios together; split unrelated logic into separate tests
Don't Forget Edge Cases
Test null, empty, boundary values, special characters
Don't Use Complex Logic in Tests
Parameterized tests should be simple assertions
